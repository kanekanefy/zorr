0.嗨,我们都是 florr 玩家 #

这份文档是给你写的 — 一个 12 岁、对 florr 烂熟于心的玩家。我们假定你知道 Mythic、Super、Pharaoh's Crown、Yggdrasil、Hel、 Mecha Spider 这些名字是什么意思,不再赘述。

但有一些"幕后机制"你可能没刻意想过 — 为什么 florr 要把 Eternal 设计成 0.1% 概率?为什么 Unique 别人能"偷"?为什么 Hel 强制 PvP?这些都跟"怎么让你玩更久 / 花更多钱"有关。这份文档把这些拆开给你看, 然后告诉你我们做的 zorr 是怎么反着设计的。

最重要的是:zorr 是你的。你能改任何数值、任何机制、加新东西、删旧东西。文档最后教你怎么动手 — 你只需要会跟 AI 描述你要什么,AI 会帮你把代码写出来、推上线。

1.三段话搞懂 zorr 是什么 #

zorr 是基于一个叫 gardn 的开源 florr 复刻项目改的(GitHub: trigonal-bacon/gardn,AGPL-3.0 协议)。这个原始 gardn 是有人独立写了一遍 florr 早期版本的核心玩法,因为 florr 本身闭源,但游戏本身没法律保护它的"玩法",所以大家可以自由复刻。

我们(zorr)在 gardn 基础上加了几个"补丁"(就是改动文件),改了标题、改了 WebSocket 连接逻辑(让镜像能跨域名部署)、改了地图底色,并且加了一个"OP Mode"补丁 — 把所有数值都改成"爽快版":升级飞快、血厚、伤害高、掉率高、怪很多、所有装备槽一开局就开。

整个游戏跑在一台 VPS 上 (https://zorr.inglegames.com),进去就能玩。没有账号,没有付费,没有时间限制。代码完全公开在 github.com/kanekanefy/zorr。

4.florr 的"氪金引擎"是怎么吃你时间的 #

这一节给你看 florr 设计师的"小心机"。理解这些 = 理解我们 zorr 为什么是反着做的。

4.1 Crafting 的概率瀑布

从 Common 升到 Eternal,每一级单次成功率减半,最后一档 Super → Eternal 是 0.1%。

算一下:平均要凑齐多少基础(Common)花瓣才能合成出最高的 Eternal?

稀有度平均所需 Common体感
Unusual6一会儿就有
Rare66玩两小时
Epic1,200玩一周
Legendary40,500玩一个月
Mythic2,600,000玩几年
Ultra340,000,000"基本不可能"
Super88,000,000,000人生不够长
Eternal220,000,000,000,000220 万亿

数字是概率倒数累乘 — 不靠氪金/挂机/抱大腿,普通玩家这辈子也合成不出 Super 以上。所以 florr 商店里能买现成的高稀有度 petal,这才是它的真正商业模式。

4.2 Unique 偷取机制

你花了大量时间凑出一个 Unique 的 Pharaoh's Crown。结果另一个玩家在某个服务器上 forge 出同样的 Unique, 系统就把你的某个 Super 消掉。你被"偷"了,而且全球范围内任何人都可能触发,你没法防御。

从游戏设计角度,这是制造稀缺 + 社交焦虑 — 让你不得不上线、不得不持续 grinding 才能"守住"装备。 这是 mobile 手游"PVP 排行 + 资源限时"的同一套话术。

4.3 Hel 强制 PvP

Hel 是 florr 唯一强制 PvP 的区域。设计目的:让你时刻警觉,**永远没有"安心练级"的环境**。也是制造焦虑的工具。

对家庭服来说有趣的部分是"家人之间互打"可以是娱乐 — 但 florr 那种"陌生人随时来宰你"完全没意义。

5.爽玩版三原则 #

原则 1:爽 over 磨。任何"为了让玩家玩更久"的设计 — crafting 概率、forging 稀缺、长 cooldown — 我们都反着做。OP 模式 = 默认开启。

原则 2:多样 over 稀缺。"难得一见的 boss"不爽,"满地都是 boss"才爽。"100 个 petal 99 个见不到"是 florr 的问题,我们 51 个全送给你,你想试哪个试哪个。

原则 3:本地 over 全球。家庭服 2-3 个人玩,排行榜上不需要 10 万人。leaderboard 只显示你和家人就够。所有"全球同服"的设计(crafting 全球通知、forge 偷取、世界排名)对我们都是噪音。

7.怎么实际改一个东西(完整流程) #

下面是从"我想改个数值"到"线上看到效果"的完整路径,大约 5-10 分钟。

  1. 找到要改的地方(下面"配方书"一节有具体位置表)
  2. 跟 AI 描述你想改什么(直接说人话:"我想让 Easy 区的 ladybug 血量翻倍")
  3. AI 帮你写一个新的 patch 文件,放进 patches/ 目录
  4. 把改动 commit + push 到 GitHub(git push origin main)
  5. GitHub Actions 自动 build 新版游戏(~3-5 分钟)
  6. 新镜像自动推到 Dokploy,Dokploy 自动重启服务器(~30 秒)
  7. 浏览器刷新 https://zorr.inglegames.com,你的改动生效

关键概念:为什么用"补丁"而不直接改源码

原始 gardn 项目在 upstream/gardn/ 目录,我们当 git submodule 锁死不动。

我们的所有改动都写成补丁文件(patches/0004-op-mode.patch 这种), 放在 patches/ 目录。Dockerfile 在 build 时按文件名顺序自动应用。

好处:gardn 上游升级时,我们的"自己写的部分"和"上游的部分"一目了然。 rebase 时知道哪些是我们的内容,不会冲突或丢失。

8.改变配方书 — 跟着抄就行 #

配方 1 · 调 OP 倍率(最简单)

小 立竿见影 涉及 1 个文件

目标:让升级更快、或更慢、或让伤害更高。

动哪:patches/0004-op-mode.patch 里的 Shared/OpMode.hh 部分。

当前默认值:

LEVEL_DIV = 10.0f      // 升级 XP / 10 (越大升级越快)
XP_MULT = 5.0f         // 每杀给 XP × 5
HP_MULT = 2.0f         // 玩家血 × 2
PETAL_DMG_MULT = 3.0f  // 打怪伤害 × 3
MOB_DMG_MULT = 0.5f    // 怪打你伤害 × 0.5
DROP_MULT = 5.0f       // 掉率 × 5
DENSITY_MULT = 3.0f    // 怪密度 × 3
ALL_SLOTS_FROM_START = true  // 8 槽全开

给 AI 说什么(直接复制改改):

"修改 patches/0004-op-mode.patch,把 OpMode 里的:
  - LEVEL_DIV 改成 50(升级再快 5 倍)
  - PETAL_DMG_MULT 改成 10(无脑碾压 mode)
  - DENSITY_MULT 改成 5(更多怪)
然后 push 上线"

AI 会帮你算好新的 patch、commit、push。等 5 分钟就生效。

配方 2 · 加一个全新 zone(纯数据扩展,没代码逻辑)

小 ★★★★ 爽 涉及 1 个文件

目标:在地图上加一块新区域,比如"Square 农场"— 几乎只刷稀有的 kSquare, 数量极少但每只给 10000 XP,孩子打到一只兴奋一晚。

动哪:Shared/StaticData.hh 的 MAP_DATA 数组。 现在的 4 个 zone 长这样:

inline std::array const MAP_DATA = std::to_array<struct ZoneDefinition>({
    { .left=0,     .top=0, .right=10000, .bottom=4000,
      .density=1, .drop_multiplier=0.3,
      .spawns = {
        { MobID::kRock, 500000 },
        { MobID::kLadybug, 100000 },
        ...
      },
      .difficulty=0, .color=0xff1ea761, .name="Easy" },
    ...
});

加一个新 zone 长这样:

{ .left=40000, .top=0, .right=50000, .bottom=4000,
  .density=0.05,            // 很稀疏,每只都是宝
  .drop_multiplier=2.0,     // 掉率也高
  .spawns = {
    { MobID::kSquare, 1000 },        // 几乎全是 Square
    { MobID::kMassiveLadybug, 10 },  // 偶尔一只巨型
  },
  .difficulty=3,
  .color=0xff8a2c5e,         // 紫红色,显眼
  .name="Square Farm" },

注意:加 zone 后,ARENA_WIDTH(在 StaticDefinitions.hh)也要相应扩大到 50000。 AI 会帮你处理这个连带改动。

给 AI 说:

"在 zorr 里加一个新 zone 叫 'Square Farm':
  - 在地图最右边(x 40000-50000)
  - 几乎只刷 kSquare(权重 1000),偶尔 kMassiveLadybug(权重 10)
  - 密度 0.05(很稀疏)
  - 掉率乘数 2.0
  - 难度 3
  - 颜色紫红色(0xff8a2c5e)
帮我生成补丁文件 patches/0005-square-farm-zone.patch,
注意 ARENA_WIDTH 也要从 40000 扩到 50000"

配方 3 · 改某只 mob 的数值

小 涉及 1 个文件

目标:比如让 Hornet 变得没那么吓人,血量减半、伤害降到 1/3。

动哪:Shared/StaticData.cc 里的 MOB_DATA 数组,每只 mob 长这样:

{
  .name = "Hornet",
  .description = "...",
  .rarity = RarityID::kRare,
  .health = { ... },     // 血量(可能是范围)
  .damage = ...,         // 撞击伤害
  .radius = { ... },     // 体型
  .xp = ...,             // 给的 XP 奖励
  .drops = { ... },      // 死后掉什么 petal
  .attributes = { ... }, // AI 行为
},

给 AI 说:

"修改 MOB_DATA 里 Hornet 的 .health 改成原来一半,.damage 改成原来 1/3,
其他不变。生成 patch 文件 patches/0005-nerf-hornet.patch"

配方 4 · 改某个 petal 的数值

小 涉及 1 个文件

目标:比如把 Basic petal 改成超强,让"什么都没装备"也能玩。

配方 5 · 万能 prompt 模板(跟 AI 协作的标准句式)

必读 跟任何配方搭配

跟 AI(像 Claude Code 或 ChatGPT)协作时,这个模板能减少 90% 误解:

背景:我在维护一个开源 florr 私服叫 zorr,代码在 github.com/kanekanefy/zorr
原始游戏代码在 upstream/gardn/ 目录(submodule,不改动)
我们的所有自定义都通过 patches/NNNN-*.patch 文件实现

我想做的改动:[这里描述你想要的效果,用人话]

请帮我:
1. 在 upstream/gardn/ 里实际修改文件实现这个效果
2. 用 git diff 生成新的 patch 文件保存到 patches/NNNN-描述.patch
3. 用 git checkout . 还原 upstream/gardn 的工作树
4. 提醒我跑 git add patches/ && git commit && git push 上线

注意:文件改完一定要 git diff 生成 patch 再还原 — 不要直接 commit upstream/gardn 的改动

为什么这个模板重要:gardn 是 submodule(从外部仓库引入的),我们不能直接 commit 它的改动 — 否则会污染上游引用。所有改动必须以 patch 形式落到我们自己的仓库里。这个模板把这个规则讲清楚了。

9.千万不要做这些事(踩坑警告) #

❌ 不要做 crafting 系统。它是 florr 的赌博引擎,故意把成功率压到 0.1%。家庭服里你直接给玩家高级 petal 就行,何苦让他们摇 5000 次?

❌ 不要做 Unique 偷取。这是 florr 用来制造社交焦虑的机制。家人之间互偷只会闹翻。

❌ 不要做账号 + 持久化收藏。私服上线下线频繁,改了数值要重启,搞账号反而麻烦。每局新鲜开始才爽。

❌ 不要做"防作弊"。私服你就是那个"作弊的人"。OP Mode 就是合法作弊。 任何反作弊代码都是浪费精力。

❌ 不要直接在 upstream/gardn/ 里 commit 修改。 所有改动通过 patches/ 走。如果你不小心 commit 了 submodule 内容, git pull 时会冲突,而且改动对上游不可见,反而难维护。

⚠️ 改 ARENA_WIDTH / ARENA_HEIGHT 要小心。 扩大地图后所有 zone 的边界要重算,不然玩家会跑到"虚空"区。 用 AI 协作时让它检查 MAP_DATA 的所有 .right / .bottom 都在新边界内。

⚠️ 加新 MobID 比想象的工作量大。 需要同时改 enum、MOB_DATA、MOB_DROP_CHANCES、RenderMob 渲染、可能还有 AI 行为。 "加 1 个新 mob" 不如"加 10 个新 zone 用现有 mob 新组合"省力。

10.快速开始 checklist #

  1. 玩一晚上现在的 https://zorr.inglegames.com,记下"哪些 mob 想强化/削弱"、"哪些 petal 觉得没用想砍"
  2. 用配方 1 的模板,让 AI 帮你调一波 OpMode 倍率(几小时就能上线一版)
  3. 用配方 3/4 调几个具体 mob/petal 的数值(每天可以推 1-2 版)
  4. 玩腻了 → 用配方 2 加一个新 zone(每周可以加 1-2 个)
  5. 想要队伍玩法 → 跟 AI 说"按 spec 里 Phase 3 的描述,实现 Squads 功能"
  6. 想要 bot 玩家陪你 → 跟 AI 说"按 spec 里 Phase 3,实现简单 bot 玩家"

12.最后 #

zorr 是你的玩具。任何不爽的数值、任何想加的功能 — 跟 AI 说一声,5 分钟后就能上线。

这种"我说什么就改什么"的体验,在 florr 这种闭源游戏里你永远得不到。 你只能等官方更新、接受官方设计、为官方花钱。
